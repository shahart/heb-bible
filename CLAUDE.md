## Project Overview

A polyglot Hebrew Bible search and analysis tool with multiple deployment targets. The same core logic (verse/pasuk search, gematria, dilugim/skip patterns) is implemented in Java (Spring Boot, several cloud function adapters), Python, Go, Rust, JavaScript (client-side), C++, and Pascal.

## Build & Run Commands

### Spring Boot (primary backend — `sb3/`)
```bash
mvn --file sb3/pom.xml spring-boot:run          # Run locally on port 8080
mvn -f sb3/pom.xml -T1C clean package           # Build jar
mvn -f sb3/pom.xml test                         # Run tests
./gradlew clean build                            # Gradle alternative (repo root)
```

### Cloud function modules (each is standalone)
```bash
cd aws-lambda && mvn clean package shade:shade   # AWS Lambda fat jar
cd oracle-cloud-fn && fn start && fn --verbose deploy --app java-app --local
```

### Other languages
```bash
# Python
cd python && python3 -m venv env && source env/bin/activate
pip install -r requirements.txt && python3 app.py
pytest tests

# Go
cd go && go build ./... && go test ./...
go run main.go שחר true

# Rust
cd rust && cargo build --verbose
```

### Docker
```bash
docker build -t hebbible-app .
docker run -p 8080:8080 hebbible-app
```

## Architecture

### Module structure
Each cloud deployment target is a **self-contained Maven module** with its own `pom.xml`. There is no single root `pom.xml` governing all modules. The canonical implementation lives in `sb3/`; cloud adapters follow the same patterns.

| Module | Purpose |
|--------|---------|
| `sb3/` | Spring Boot 3 REST API (primary) |
| `aws-lambda/` | AWS Lambda handler |
| `spring-cloud-function/` | Spring Cloud Function (AWS-compatible) |
| `gcp-cloud-function/` | Google Cloud Functions |
| `azure-functions/` | Azure Functions |
| `alibaba-function-compute/` | Alibaba Function Compute |
| `tencent/` | Tencent Cloud |
| `oracle-cloud-fn/` | Oracle Cloud (via `fn` CLI) |
| `javafx/` | Desktop GUI |
| `client/`, `docs/` | Client-side JavaScript |
| `go/`, `python/`, `rust/`, `c++/`, `pascal/` | Language samples |

### Data layer
All implementations load the same data:
- `bible.txt.gz` — primary source: gzipped CSV (`reference,hebrew_text`) with 22,329 verses
- Legacy: `bible.txt.5bit` (DOS 5-bit encoding, code page 862)
- Fetched from GitHub on startup; cached in memory after first load

### Core data flow (Spring Boot)
```
POST /psukim {name}
  → Controller → ServiceImpl.psukim()
  → Repo.init() (loads bible.txt.gz once)
  → filter verses: first char == name[0] AND last char == name[-1]
     (or substring match if containsName=true)
  → optional DynamoDB logging
  → return List<Pasuk> (book, bookNo, perek, pasuk, text, cntLetter)
```

REST endpoints: `POST /psukim`, `POST /dilugim`, `GET /psukim` (returns total count).

### Handler names (Lambda)
- Standalone Lambda: `edu.hebbible.Handler::handleRequest`
- Spring Cloud Function: `org.springframework.cloud.function.adapter.aws.FunctionInvoker::handleRequest`

## Key Conventions

- **Never edit `dependency-reduced-pom.xml`** — generated by the Maven shade plugin.
- Java version is 21 across modules; change via the `java.version` property in the relevant `pom.xml`.
- For cross-cutting changes: update `sb3/` first, then adapt each cloud adapter module.
- Hebrew letter normalization (final forms → base forms: ם→מ, ן→נ, etc.) happens during data load in `Repo`.
- DynamoDB logging is optional and gated by the `prod` Spring profile; tests use the `test` profile to skip it.
- `gematria` is a custom Maven artifact (`edu.gematria:gematria:1.0`) used as a library dependency.
